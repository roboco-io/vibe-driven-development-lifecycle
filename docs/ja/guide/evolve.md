# ステージ6 — 還流 (Evolve)

> 還流のないVDLCは、ただの速いコーディングだ。

## このステージの目的

ステージ5でデプロイし観察した結果は、それ自体では何も変えない。このステージの役割は、サイクルで学んだことを次サイクルがより良く回るようコンテキスト資産へ還元することだ。原則4(コンテキストは資産だ)が言うように、VDLCはサイクルを回すたびに資産が厚くなり、資産が厚くなるほど次サイクルが速くなる複利構造で設計されている。このステージを飛ばすと、毎サイクルが最初から同じ誤りを繰り返す、ただ速いだけのコーディングに堕する。還流の対象は組織の資産だけではない。原則6(理解が所有だ)が言うように、個人の理解もここでともに育たねばならない。

## 実行順序

1. **繰り返された指示をプロジェクトルールへ移す。** 同じ指示を二度以上繰り返したなら、それはエージェントが自ら知っておくべきルールだ。CLAUDE.mdやこれに準ずるルールファイルへ追加する。
2. **検証で捕まえた誤りのパターンをレビューチェックリストに反映する。** ステージ4(検証)で同じ種類の問題が繰り返し見つかったなら、その問題を漉し取る項目をチェックリストに追加し、次サイクルからは自動で捕まるようにする。
3. **新たに整理されたドメイン知識をwikiへ移す。** サイクルを進めながら明らかになったドメインルール、用語、例外ケースは、対話ログに埋もれさせずwiki文書として整理する。
4. **理解しないまま通り過ぎた点を償還する。** サイクルで初めて出会ったパターンや技術を振り返って自分の言葉で整理し、承認はしたが説明できないコードがあれば、ここで学習して認知負債を返す。
5. **サイクルの振り返りを軽く、時間単位で回す。** スプリントの振り返りのように週単位でまとめてやらない。サイクルが終わるたびに数分あれば十分だ — 何を繰り返し指示したか、何が検証で引っかかったか、何を新しく学んだか、三つの問いだけを見渡す。

## 成果物

- 更新されたプロジェクトルール、レビューチェックリスト、wiki文書 — 最低一つ以上
- サイクルの振り返り記録 — 三つの問いへの短い答えで十分だ

還流が実際に組織の速度を引き上げているかは[導入指標](/ja/adoption/metrics)で追跡する。

## 完了基準チェックリスト

- [ ] 今サイクルでコンテキスト資産(ルール・チェックリスト・wiki)のうち一つ以上が更新された
- [ ] 繰り返された指示があったならプロジェクトルールに反映された
- [ ] 検証で捕まえた誤りのパターンがあったならレビューチェックリストに反映された
- [ ] 新たに明らかになったドメイン知識があったならwikiに反映された
- [ ] 理解しないまま通り過ぎた点があったなら振り返って自分の言葉で整理した
- [ ] サイクルの振り返りがスプリント単位ではなく、サイクルが終わるたびに軽く行われた

## よくある誤り

- **還流を省いて次サイクルへすぐ移る。** 学んだことを資産に反映しないと、同じ誤りが次サイクルでそっくり繰り返される。速度は速いが組織は賢くならない。
- **振り返りをスプリント単位でまとめてやる。** 数週間後にまとめて振り返ると、何を繰り返し指示したか、何がなぜ引っかかったかの細部がぼやける。サイクルが終わった直後、記憶が鮮明なうちに短くやるほうがよい。
- **資産の更新を個人メモとして残す。** 更新されたルールや知識がエージェントの読むファイル(CLAUDE.md、wiki、チェックリスト)へ入らないと、次サイクルでエージェントは相変わらず何も知らない。

## Claude Codeなら

::: tip
サイクルを終えるたびに「今回、繰り返し指示したことはあったか?」を自らに問い、あればCLAUDE.mdに一行追加する習慣をつける。複数のプロジェクトで繰り返し使う手順なら、スラッシュコマンドやスキルへ昇格させて再利用可能な資産にする。

セッションをまたぐ学習はメモリ機能で補える。特定のセッションで見つけた誤りのパターンやドメイン知識をメモリに残しておけば、次のセッションで同じ説明を繰り返さず、すぐに続けて作業できる。
:::
