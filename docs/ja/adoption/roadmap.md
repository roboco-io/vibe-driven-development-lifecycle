# 導入ロードマップ

> 組織導入は4つのステップで進める。各ステップは[成熟度モデル](/ja/adoption/maturity)の特定区間を通過する過程でもある。

VDLCは全社に一度に適用する方式で導入しない。失敗コストが低いところで六つのステージ全体を一周回してみて、そこから出た資産とルールを足場に範囲を広げていく順を勧める。各ステップの完了信号を確認した後にだけ次のステップへ移る — 信号なしに範囲だけを広げると、[マニフェスト](/ja/manifesto)が指摘したアンチパターンを組織全体に複製することになる。

## ステップ1 — パイロット

**対応成熟度**: [L1 探索](/ja/adoption/maturity) → L2 実践への転換

### 目標

失敗コストが低い領域で[六つのステージ](/ja/guide/intent)全体を最初から最後まで一度経験する。このステップの目的は速度の検証ではなく、意図定義から還流まで全区間を実際に回してみること自体だ。

### やるべきこと

- ユーザーが限定的で、失敗しても売上に直接影響しない内部ツールや新規の小規模プロジェクトを一つ選ぶ。
- [ステージ1 — 意図定義](/ja/guide/intent)から[ステージ6 — 還流](/ja/guide/evolve)まで省略なく順に進める。
- 計画承認の関門、自動検証といった最小限の仕掛けだけを立て、残りはシンプルに保つ。このステージで完璧な体系を作ろうとしない。

### 完了信号

- パイロットプロジェクトが六つのステージを一周以上完走した。
- 意図文書と計画承認なしに進めたサイクルと、手順を守ったサイクルの違いを、チームが自ら説明できる。
- 「検証なきバイブ」「コンテキストなきプロンプト」といったアンチパターンを最低一度は自ら経験し、原因を指摘できた。

## ステップ2 — コンテキスト資産化

**対応成熟度**: L2 実践 → [L3 資産化](/ja/adoption/maturity)への進入準備

### 目標

パイロットで繰り返し説明したり即興で判断したりしたルールと知識を、再利用可能な形へ整理する。このステップがないと、パイロットの経験はそのプロジェクトの中でだけ有効で、次のプロジェクトへつながらない。

### やるべきこと

- パイロット中にセッションごとに繰り返し説明したドメインルールを、プロジェクトルールファイル(CLAUDE.mdなど)へ移す。
- 検証ステージで繰り返し捕まった誤りのパターンをレビューチェックリストにする。
- リスクの異なる成果物に検証強度をどう変えて適用したかをリスク等級表として整理する。
- [ステージ6 — 還流](/ja/guide/evolve)をパイロットの最後のサイクルで一度きりにせず、今後も繰り返される習慣として根づかせる。

### 完了信号

- 次サイクルを始めるとき、以前と同じ説明を最初から繰り返さない。
- チーム外の別の人が、プロジェクトルールファイルとチェックリストを見ただけでチームの判断基準をおおよそ把握できる。
- コンテキスト資産(ルールファイル、チェックリスト、等級表)がリポにコミットされバージョン管理される。

## ステップ3 — チーム拡散

**対応成熟度**: [L3 資産化](/ja/adoption/maturity)の定着

### 目標

パイロットのルールと資産を個人所有からチーム共用へ転換し、テスト基準・レビュールール・リスク等級を備えた検証体系をチーム単位で立てる。

### やるべきこと

- コンテキスト資産を個人のリポジトリではなく、チームがともに参照し更新する場所(共有リポ、wiki)へ移す。
- リスク等級に応じて検証強度を分ける基準を、チーム全体が同じように適用するよう合意する。
- 計画承認の関門と交差レビューをチームの標準作業手順として文書化する。
- サイクルが終わるたびに学んだことを資産へ反映する還流の儀礼(定期の振り返り、チェックリスト更新)をチームの予定に入れる。

### 完了信号

- チームメンバーの大半が個人の判断ではなく、チーム共用のルールと等級表を基準に検証強度を定める。
- 新規参加者がオンボーディングの過程でチームのコンテキスト資産を参照して速く合流する。
- 還流の儀礼が特定個人の習慣ではなく、チームの予定に固定された手順として存在する。

## ステップ4 — 組織標準

**対応成熟度**: [L4 標準化](/ja/adoption/maturity)

### 目標

VDLCを特定チームの実践ではなく組織の標準プロセスとして定着させ、指標でその効果を管理する。

### やるべきこと

- [測定指標](/ja/adoption/metrics)4種(サイクルリードタイム、手戻り率、検証通過率、コンテキスト資産の増加量)を定期的に収集する体系を作る。
- チームごとに散らばっていたコンテキスト資産のうち、組織全体に有効なものを選別して共用資産へ昇格する。
- 新規プロジェクトと新規入社者のオンボーディングに、組織標準資産の再利用を基本手順として入れる。
- 指標が悪化する信号を定期的に点検し、必要なら検証体系や資産を更新する責任者を指定する。

### 完了信号

- 指標が特定チームではなく組織単位で定期収集され、意思決定に実際に使われる。
- 新しいプロジェクトが最初から組織標準のコンテキスト資産を再利用して始まる。
- VDLC適用の可否が個人の選択ではなく、組織の既定値だ。
